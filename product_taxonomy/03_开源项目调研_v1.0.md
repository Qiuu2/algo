# 开源项目与标准调研：产品分类 + 参数字典 + 参数抽取（v1.0）

> 结论先行：这个领域**确实有大量开源积累，而且分三层都能用上**——
> ① 分类+参数字典有现成的国际开放标准（ETIM），和我们设计的体系同构，可采其方法和字段命名；
> ② 管数据的开源 PIM 软件有好几个成熟项目，将来多人维护时可以直接部署；
> ③ 参数抽取环节有 Google 开源的 LangExtract、instructor 等 LLM 结构化抽取框架和单位解析库可用。
> 但**没有任何现成项目能直接吃进我们的参数表吐出成品**——分类校准、字典定义、拆解规则仍是我们自己的活；开源给的是"轮子"，不是"整车"。
> 日期：2026-07-05

## 一、分类与参数字典层：国际开放标准

### 1.1 ETIM（欧洲技术信息模型）— ★ 最对口，建议采其方法论

- 电工/电子/暖通/楼宇技术产品的国际分类标准，**开放免费**（etim-international.com，在线 Viewer 可查全部类目）；
- 结构与我们的设计**完全同构**：Product Group（≈大类）→ Product Class（≈末级分类，5500+ 个类）→ 每个 Class 挂一组 **Feature（参数字典）**，每个 Feature 规定数据类型——**A 字母（=我们的枚举/文本）、L 逻辑（=布尔）、N 数值、R 范围（=我们的 min/max 两列）**，数值型带标准单位；
- 含扬声器、功放、摄像机、LED 灯具等电子电工类目，其 Feature 命名和单位口径可直接参考进我们的字典；
- **局限**：欧洲电工批发视角，中国声光电行业特色品类（无纸化会议、IP 网络广播、录播、智慧教室）没有现成 Class；无中文官方版。

**用法**：不整体采用，但① 数据模型采标（我们的四种列类型恰与其 A/L/N/R 一一对应，等于业界已验证这套思路）；② 做功放/音箱/摄像机字典时对照其同类 Class 的 Feature 清单查漏补缺；③ 若将来产品要出海对接欧洲分销商，字典里预留 ETIM Class 映射列即可平滑接轨。

### 1.2 其他分类标准（简评）

| 标准 | 开放性 | 有无参数字典 | 结论 |
|---|---|---|---|
| eCl@ss | 收费 | 有（很全） | 不采 |
| UNSPSC | 免费下载 | **无**（纯分类码） | 仅政府采购编码需要时映射 |
| GS1 GPC | 会员制 | Brick 属性较浅 | 不采 |
| Google Product Taxonomy | 开放 CSV | 无 | 消费品视角，太浅，不采 |

## 二、数据管理层：开源 PIM（产品信息管理）系统

这类软件就是为"分类树 + 每类一套参数模板 + 产品录入/导出"而生的，概念完全对得上：Akeneo 的 **Family（产品族）= 我们的末级分类参数模板**，Category tree = 我们的分类树。

| 项目 | 技术栈 | 特点 | 顾虑 |
|---|---|---|---|
| [Akeneo CE](https://github.com/akeneo/pim-community-dev) | PHP/Symfony | 最出名，界面好，Family/Attribute 模型标准 | 社区版 2023 年起不再加新功能 |
| Pimcore | PHP | PIM+DAM+CMS 一体，属性可继承 | 重，配置要写 PHP |
| OpenPIM / AtroPIM / UnoPIM | Node/PHP | 轻量，Docker 一键起，界面配置无需编程 | 生态小 |
| NocoDB / Baserow（泛用表格库） | — | 开源 Airtable，建表灵活、API 好、上手零成本 | 无 PIM 专属概念，规范靠自律 |

**建议**：**现阶段不上 PIM**。P1/P2 阶段交付物就是 Excel/CSV + git 版本管理，够用且 AI 读取最方便。等出现"多人同时维护、新品持续录入、要接官网/商城"的需求时，再把定稿的分类树和字典灌进 UnoPIM/OpenPIM（轻量 Docker）或 NocoDB——**我们的字典表设计天然就是这些系统的导入格式**，先做表后上系统零浪费。

## 三、参数抽取层：LLM 结构化抽取工具与研究

### 3.1 可直接用的工具

- **[LangExtract](https://github.com/google/langextract)**（Google 开源，Python）：LLM 从非结构化文本抽取结构化信息，**核心卖点是 source grounding——每个抽取值精确回链到原文位置**，这正是我们"原文永不改动、拆解可溯源"的机器化实现；支持长文档分块、并行、多轮抽取，输出 JSONL。
- **[instructor](https://github.com/567-labs/instructor)** / Outlines：给 LLM 输出套 Pydantic/JSON Schema 强校验，模型吐的数不合字典定义就自动重试——等于把我们的参数字典直接变成抽取器的校验器。
- **[quantulum3](https://github.com/nielstron/quantulum3)**：从文本里抽取"数量+单位"（290+ 单位，支持范围、口语数字）；**[pint](https://github.com/hgrecco/pint)** / unit-parse：单位换算与量纲校验（kHz→Hz、g→kg 这类换算的程序化保障）。

### 3.2 研究侧佐证（不直接用，但验证路线）

- **MAVE 数据集**（Google Research，300 万属性标注/1257 类目）、ImplicitAVE：属性值抽取的学术基准，英文消费品，对我们无直接迁移价值；
- **ExtractGPT 等 2023-24 研究**：GPT-4 级 LLM 做产品属性抽取 F1 ≈ 91%，超过全部专门训练的小模型——**结论：用大模型+schema 校验做参数拆解是当前最优解，不必自己训模型**。这正是我们方法论文档里"AI 拆解 + 解析状态标注 + 人工复核推断/冲突"路线的学界背书：约 9% 的疑难就是复核清单要兜的部分。

## 四、落到我们项目的技术组合（更新后的执行方案）

| 环节 | 采用 | 开源借力点 |
|---|---|---|
| 分类树 | 自建（已有草案） | 对照 ETIM Class 查漏；预留 ETIM 映射列 |
| 参数字典 | 自建，四类型模型 | 类型体系对标 ETIM A/L/N/R；功放/音箱/摄像机字典对照 ETIM Feature 清单 |
| 拆解引擎 | LLM + 字典 schema 校验 | LangExtract（溯源）或 instructor（校验重试）二选一做流水线骨架 |
| 单位换算校验 | 脚本 | pint / quantulum3 |
| 数据载体 | 近期 Excel/CSV + git；中期可选 | UnoPIM / OpenPIM / NocoDB，字典表即导入格式 |
| 质控 | 规则脚本 + 抽检 + 复核清单 | 校验规则用字典的"允许值区间"字段驱动 |

对方法论评审稿（02 文档）的影响：**流程和产出物不变**，第五节"质量控制"的校验脚本和第四节的拆解执行可以站在这些轮子上做，开发量比从零写小得多；字典表新增可选字段 `ETIM映射`。

## 五、明确不建议的方向

1. **整体照搬任何一家标准/系统的分类树**——中国声光电行业的品类结构（IP 广播、无纸化、录播）没有哪个国际标准覆盖得好，硬套会扭曲业务；
2. **自己训练属性抽取模型**（OpenTag/MAVE 路线）——研究已证明通用大模型+schema 更强更省；
3. **一上来就部署 PIM 系统**——数据还没定型就上系统，等于把返工搬进数据库里做。
